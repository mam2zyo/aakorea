# A.A. KOREA 웹앱 도메인 명세서

## 1. 문서 목적

이 문서는 A.A. KOREA 웹앱의 아키텍처 원칙과 도메인 모델을 정의하는 기준 문서다.
프로젝트 개요는 `README.md`, 사용자 그룹별 서비스 요구사항은 `docs/REQUIREMENTS.md`에서 다루며, 본 문서는 다음 내용을 개발 기준으로 고정한다.

- 모듈러 모놀리스 기반의 시스템 경계
- 각 도메인 모듈의 책임과 데이터 소유권
- CQRS-Lite 관점의 읽기/쓰기 분리 원칙
- 스냅샷 패턴을 포함한 데이터 보존 규칙
- 향후 MSA 분리를 고려한 결합도 관리 규칙

이 문서는 화면 요구사항 문서가 아니라, 도메인 모델과 애플리케이션 구조를 정의하는 문서다.

---

## 2. 문서 범위

본 문서는 다음 질문에 답하기 위한 문서다.

- 시스템을 어떤 모듈로 나눌 것인가
- 각 모듈은 어떤 데이터를 소유하는가
- 읽기와 쓰기 책임은 어떻게 분리되는가
- 주문, 구독, 변경 이력 같은 과거 사실은 어떻게 보존하는가
- 향후 서비스 분리를 고려할 때 지금 어떤 제약을 지켜야 하는가

반면 다음 내용은 본 문서의 주범위가 아니다.

- 사용자별 상세 화면 요구사항
- 메뉴/콘텐츠 운영 정책의 세부 문안
- 페이지 단위 UX 흐름

이 항목들은 `docs/REQUIREMENTS.md`에서 우선 정의하고, 필요 시 별도 화면 설계 문서로 확장한다.

## 3. 아키텍처 원칙

### 3.1. 모듈러 모놀리스

시스템은 단일 애플리케이션으로 시작하지만, 내부는 비즈니스 도메인별로 강하게 분리된 모듈러 모놀리스 구조를 채택한다.

- 하나의 배포 단위 안에서 개발/운영 복잡도를 낮춘다.
- 도메인별 패키지 경계를 엄격히 유지해 변경 영향을 국소화한다.
- 향후 `basic`, `service`, `store`, `heart`를 별도 서비스로 분리할 수 있도록 설계한다.

권장 최상위 모듈은 다음과 같다.

- `modules/basic`
- `modules/service`
- `modules/member`
- `modules/store`
- `modules/heart`

공통 기술 요소는 도메인 하위가 아니라 별도 공통 영역에서 제공하되, 공통 영역이 도메인 규칙을 침범하지 않도록 제한한다.

### 3.2. 도메인 중심 패키징

패키지는 `controller / service / repository` 같은 기술 계층 중심이 아니라, 모듈과 하위 서브도메인 중심으로 구성한다.

예시:

```text
modules/
  basic/
    group/
    meeting/
    notice/
  service/
    district/
    groupadmin/
    gsr/
  member/
    auth/
    profile/
    address/
  store/
    item/
    cart/
    order/
  heart/
    subscription/
    delivery/
    issue/
```

기술 계층은 각 도메인 내부로 캡슐화한다.

### 3.3. CQRS-Lite

본 프로젝트는 전면적인 이벤트 소싱/별도 읽기 저장소 기반 CQRS가 아니라, 실용적인 CQRS-Lite를 지향한다.

- 일반 사용자 대상 조회 기능은 `basic` 모듈에서 읽기 최적화 관점으로 제공한다.
- 관리자/봉사자 대상 생성·수정·삭제 기능은 `service` 모듈에서 명령 중심으로 제공한다.
- 동일한 개념이라도 읽기 모델과 쓰기 모델은 분리된 API/DTO/유스케이스를 가질 수 있다.
- 필요한 경우 동일 테이블을 읽더라도, 조회 전용 projection이나 집계 DTO를 분리한다.

예시:

- `basic`은 그룹 상세 화면에서 그룹 기본 정보, 모임 일정, 활성 공지를 한 번에 조합해 반환한다.
- `service`는 그룹 수정, 모임 생성/수정/삭제, 공지 등록, 승인 상태 변경 같은 명령을 처리한다.

### 3.4. 느슨한 결합과 직접 참조 금지

모듈 간 엔티티 참조는 최소화한다. 특히 `store`와 `heart`는 `member` 모듈의 JPA 엔티티를 직접 연관관계로 참조하지 않는다.

- 허용: `memberId`, `createdByMemberId`, `buyerMemberId` 같은 식별자 값 참조
- 금지: `@ManyToOne Member member` 형태의 직접 연관

이 원칙은 다음 이점을 가진다.

- 모듈 간 런타임 결합도 감소
- 서비스 분리 시 데이터베이스 경계 이동이 쉬움
- 주문/구독과 회원 마스터 간 생명주기 분리 가능

### 3.5. 스냅샷 패턴

결제, 주문, 구독처럼 거래 시점의 사실을 보존해야 하는 영역에서는 원본 마스터 데이터에 의존하지 않고 텍스트 스냅샷을 저장한다.

- 주문 시점의 배송지 정보는 `Order`에 복사 저장한다.
- 주문 시점의 판매 가격은 `OrderItem`에 복사 저장한다.
- 구독 시점의 수령인 정보는 `Subscription`에 복사 저장한다.
- 이후 회원 주소록이 바뀌더라도 과거 주문/결제 기록은 변경되지 않는다.

이는 감사 추적, CS 대응, 세무/운영 기록 보존에 필수다.

### 3.6. 변경 이력과 운영 감사 가능성

운영 데이터는 "현재 상태"만 남기지 않고, 중요한 변경은 감사 가능해야 한다.

- 그룹 정보 변경은 `GroupChangeLog`에 기록한다.
- 필요 시 변경 전/후 JSON 스냅샷을 남긴다.
- 수정자, 수정 시각, 변경 사유를 추적 가능하게 설계한다.

## 4. 공통 도메인 개념

### 4.1. 주요 역할(Role)

- `USER`: 일반 회원
- `GSR`: 그룹 대표 봉사자
- `ADMIN`: 전체 운영 관리자

역할은 인증/인가의 기준일 뿐, 모든 비즈니스 책임을 역할 자체에 밀어 넣지 않는다. 실제 권한은 모듈별 유스케이스 단위로 정의한다.

### 4.2. 핵심 식별자

- 모든 영속 엔티티는 기본적으로 Long 기반 식별자를 사용한다.
- 외부 노출용 식별자는 필요 시 별도 공개용 키를 둘 수 있으나, 내부 연관은 Long 기준으로 통일한다.
- 모듈 간 참조는 식별자 값만 사용한다.

### 4.3. 주요 공통 개념

- `Province`: 사용자 친화적인 행정구역 검색 기준
- `District`: A.A. 내부 운영 조직 단위
- `MeetingType`: `OPEN`, `CLOSED`, `MIX`
- `MIX`는 공개/비공개를 오가며 운영하는 정기 모임을 표현하는 타입이며, 구체적인 운영 방식은 모임 소개나 공지로 보완한다.
- `OrderChannel`: `ONLINE`, `OFFLINE`

`Province`와 `District`는 반드시 분리한다. 하나는 사용자 검색 편의용 지리 기준이고, 다른 하나는 내부 조직 운영 기준이다.

| --------- | ------------------------------------- | -------------------- | ---------------- |

## 6. Basic 모듈 명세

### 6.1. 목적

`basic` 모듈은 비로그인 일반 사용자가 A.A. 정보를 얻고, 자신에게 맞는 모임을 빠르게 찾는 읽기 전용 핵심 도메인이다.

### 6.2. 핵심 엔티티/읽기 모델

- `Group`
- `Meeting`
- `MeetingPlace`
- `GroupNotice`

여기서의 `Group`, `Meeting`은 조회 관점의 모델이다. 쓰기 책임은 `service` 모듈이 가진다.

### 6.3. 주요 기능

#### 6.3.1. 다중 조건 모임 검색

사용자는 다음 조건을 조합하여 모임을 검색할 수 있어야 한다.

- `Province`
- `DayOfWeek`

결과는 기본적으로 시간순 정렬을 제공한다.

대표적인 조회 시나리오:

- 서울특별시의 화요일 모임 목록
- 특정 지역의 오늘 모임 목록

#### 6.3.2. GIS 기반 검색

`MeetingPlace`는 위도/경도 좌표를 보유할 수 있으며, PostgreSQL + PostGIS를 사용해 다음 기능을 지원한다.

- 지도 시각화
- 현재 위치 기준 주변 모임 검색
- 특정 반경 내 모임 조회

지도 UI는 카카오맵 또는 네이버맵 연동을 전제로 한다.

#### 6.3.3. 그룹 상세 통합 조회

특정 그룹 상세 페이지는 최소 다음 정보를 한 번에 제공해야 한다.

- 그룹 기본 정보
- 정기 모임 일정
- 현재 활성화된 공지사항
- 연락 정보와 장소 정보

프론트엔드는 이 집계 응답을 사용해 추가 조합 없이 상세 화면을 렌더링할 수 있어야 한다.

#### 6.3.4. 정적/반정적 콘텐츠 제공

초심자 및 일반 방문자를 위한 콘텐츠를 제공한다.

- 처음 오신 분들께
- A.A. 소개
- 자가진단
- 첫 모임 가이드
- 익명성 및 참여 안내

이 콘텐츠는 완전한 CMS 수준이 아닐 수 있으나, 운영자가 관리 가능하거나 배포 가능한 구조를 고려한다.

### 6.4. 읽기 모델 설계 원칙

- `basic`은 응답 속도와 사용성 중심으로 설계한다.
- 복잡한 관리자 내부 상태보다는, 사용자에게 필요한 현재 공개 상태만 노출한다.
- 비공개/중지/만료 정보는 조회 결과에서 명확히 제외하거나 별도 상태 규칙을 둔다.

### 6.5. 예시 쿼리 원칙

`README.md`에서 정의한 방향처럼, 서버는 특정 지역과 요일 기준의 모임을 시간순으로 빠르게 반환할 수 있어야 한다.

```java
List<Meeting> findByGroup_ProvinceAndDayOfWeekOrderByStartTimeAsc(
    Province province,
    DayOfWeek dayOfWeek
);
```

실제 구현에서는 공개 상태, 활성 공지 여부, 장소 좌표 유무 등도 함께 고려한 조회 모델이 필요할 수 있다.

## 7. Service 모듈 명세

### 7.1. 목적

`service` 모듈은 A.A. 조직망과 그룹의 최신 정보를 유지하기 위한 쓰기 중심의 백오피스 도메인이다.
관리자와 GSR이 사용하는 운영 기능의 중심이 된다.

### 7.2. 핵심 엔티티

- `District`
- `Group` (Command 모델)
- `Meeting` (Command 모델)
- `GSR`
- `GroupChangeLog`
- `Notice` 또는 `GroupNotice`의 관리 모델

### 7.3. 주요 기능

#### 7.3.1. 조직(District) 관리

- 지역연합 생성, 수정, 삭제
- 통폐합 및 명칭 변경
- 그룹과 지역연합 매핑 관리

초기 기준으로 11개 지역연합을 운영하되, 개수는 고정 상수가 아니라 데이터로 관리한다.

#### 7.3.2. 권한 및 봉사자(GSR) 관리

- GSR 계정 생성
- 역할 승인/정지
- 담당 그룹 연결
- 계정 상태 관리

GSR는 장기적으로 그룹 정보를 직접 수정할 수 있는 운영 주체다.

#### 7.3.3. 그룹 및 모임 관리

- 그룹 기본 정보 수정
- 설립일, 연락처, 안내 정보 관리
- 정기 모임 추가/수정/삭제
- 모임별 요일, 시간, 종류, 상태 관리
- 장소 정보 및 온라인/오프라인 속성 관리

#### 7.3.4. 공지 관리

그룹별 임시 운영 이슈를 공지로 관리한다.

- 임시 휴무
- 장소 변경
- 특별 행사 안내
- 팝업/배너형 노출
- 노출 시작/만료일 설정

공지의 공개 여부와 노출 기간은 `basic` 조회 결과에 직접 반영된다.

#### 7.3.5. 변경 이력 관리

그룹 핵심 정보가 변경될 때는 `GroupChangeLog`를 생성한다.

최소 기록 항목:

- 변경 대상 그룹 ID
- 수정자 ID
- 수정 일시
- 변경 사유
- 변경 전 데이터
- 변경 후 데이터

저장 포맷은 JSON 스냅샷이 가장 유력하며, 롤백보다도 "무엇이 왜 바뀌었는가"를 추적하는 목적이 우선이다.

### 7.4. Command 모델 설계 원칙

- 입력 검증은 강하게, 조회 조합은 `basic`으로 위임한다.
- 쓰기 API는 성공/실패가 명확해야 하며, 상태 전이는 도메인 규칙으로 제한한다.
- 공개 노출에 영향을 주는 필드는 변경 이력을 남긴다.

## 8. Member 모듈 명세

### 8.1. 목적

`member` 모듈은 시스템 전반의 사용자 인증과 공통 마스터 데이터 관리를 담당한다.

### 8.2. 핵심 엔티티

- `Member`
- `DeliveryAddress`

### 8.3. 주요 기능

#### 8.3.1. 인증/인가

- JWT 기반 로그인
- 역할 기반 접근 제어
- 사용자 상태 관리

인증은 공통 인프라이지만, 역할 검증은 각 모듈 유스케이스 안에서 다시 점검해야 한다.

#### 8.3.2. 회원 프로필 관리

- 기본 프로필 정보 관리
- 역할/상태 조회
- 연락처 및 운영용 속성 관리

#### 8.3.3. 주소록 마스터 관리

- 기본 배송지 설정
- 다중 배송지 저장
- 주소 수정/삭제

이 주소록은 `store`, `heart`에서 결제 직전 선택용 원본 데이터로만 사용한다.

### 8.4. 다른 모듈과의 관계

- `store`는 주문 생성 시 `memberId`와 주소록 원본을 조회해 스냅샷을 만든다.
- `heart`는 구독 생성 시 `buyerMemberId`와 수령인 주소 정보를 분리 저장한다.
- `service`는 운영 계정 권한 부여 시 `Member`의 역할을 조정할 수 있다.

직접 연관관계 대신 식별자와 DTO 기반 조회를 사용한다.

## 9. Store 모듈 명세

### 9.1. 목적

`store` 모듈은 A.A. 공식 교본, 팸플릿 등 실물 문헌을 판매하는 단발성 이커머스 도메인이다.
향후 `store.aakorea.org`로 분리 가능한 구조를 목표로 한다.

### 9.2. 핵심 엔티티

- `Item`
- `Cart`
- `CartItem`
- `Order`
- `OrderItem`

### 9.3. 주요 기능

#### 9.3.1. 상품 및 재고 관리

- 상품 코드 관리 예: `B-1`
- 카테고리 관리
- 가격 관리
- 재고 수량(`stockQuantity`) 관리
- 판매 가능 여부 관리

재고는 실물 기준이며, 디지털 상품을 전제로 하지 않는다.

#### 9.3.2. 장바구니 및 온라인 주문

- 사용자는 장바구니에 상품을 담는다.
- 주문 생성 시 장바구니 항목을 주문 항목으로 확정한다.
- PG 결제를 통해 `ONLINE` 주문을 생성한다.

결제 이전의 장바구니 가격과 결제 시점의 확정 가격은 다를 수 있으므로, 주문 시 가격 스냅샷을 생성한다.

#### 9.3.3. 오프라인 주문 등록

관리자는 행사/현장 판매 등 오프라인 판매를 `OFFLINE` 채널로 직접 등록할 수 있다.

- 즉시 재고 차감
- 결제 완료 상태로 시작 가능
- 별도 배송이 없는 경우 수령 완료 처리 가능

#### 9.3.4. 스냅샷 기반 주문 생성

`Order`는 결제 시점의 배송지 스냅샷을 보존한다.

예시 필드:

- 수령인명
- 연락처
- 우편번호
- 기본 주소
- 상세 주소
- 배송 메모

`OrderItem`은 결제 시점의 가격과 상품명을 보존한다.

예시 필드:

- itemId
- itemCodeSnapshot
- itemNameSnapshot
- unitPriceSnapshot
- quantity

#### 9.3.5. 주문 상태 관리와 배송 처리

대표 상태 예시:

- `PENDING_PAYMENT`
- `PAID`
- `SHIPPED`
- `CANCELLED`

운영자는 `PAID` 주문을 기준으로 배송지 스냅샷을 엑셀로 추출해 택배 라벨을 출력하고, 발송 후 `SHIPPED`로 변경한다.

### 9.4. 도메인 규칙

- 결제 완료 이후 주소록 마스터 변경이 기존 주문을 바꾸지 않는다.
- 결제 완료 이후 가격 변경이 기존 주문 금액을 바꾸지 않는다.
- 재고 차감 시점은 온라인은 결제 확정, 오프라인은 등록 확정 시점이 기본이다.
- 주문 취소 시 재고 복구 규칙은 명시적으로 설계한다.

## 10. Heart 모듈 명세

### 10.1. 목적

`heart` 모듈은 월간지 장기 구독, 발송 이력, 만료 관리, 디지털 아카이브를 담당하는 정기 구독 도메인이다.
향후 `heart.aakorea.org`로 분리 가능한 구조를 목표로 한다.

### 10.2. 핵심 엔티티

- `Subscription`
- `DeliveryHistory`
- `MagazineIssue`

### 10.3. 주요 기능

#### 10.3.1. 장기 구독

- 1년(12회), 2년(24회) 등 선결제 기반 플랜
- 플랜별 총 발송 횟수 보유
- 구독 시작일, 종료 예정일, 잔여 횟수 계산

#### 10.3.2. 선물하기(Gift)

결제자와 수령인이 다를 수 있다.

- `buyerMemberId`는 결제자 식별자
- 수령인은 회원일 수도, 비회원일 수도 있다
- 수령인 정보는 스냅샷 텍스트로 저장한다

교도소, 병원, 시설 등 특수 수령처도 지원해야 하므로, 일반 회원 주소록 모델에 종속되지 않는다.

#### 10.3.3. 구독 배송지 변경

구독 도중 주소 변경이 발생하면 `Subscription` 내부의 수령인 스냅샷을 수정한다.

중요한 점:

- 과거 발송 이력은 당시 기준 사실로 남는다.
- 현재 활성 구독의 다음 발송 대상만 최신 스냅샷을 따른다.

#### 10.3.4. 발송 횟수 관리

매월 발송 명단을 추출할 때 다음이 발생한다.

- `DeliveryHistory` 생성
- `Subscription.sentCount` 증가
- 필요 시 마지막 발송 일시 기록

`sentCount >= totalIssues`가 되면 구독 상태를 `EXPIRED`로 자동 전환한다.

#### 10.3.5. 만료 임박 알림

배치 작업 또는 스케줄러로 다음 대상을 조회한다.

- 잔여 1회 구독자
- 또는 만료 30일 전 구독자

이후 연장 안내 메시지를 발송한다.

알림 채널은 추후 확정하되, 문자/이메일/앱 알림 중 하나 이상을 고려한다.

#### 10.3.6. 디지털 아카이브

`MagazineIssue`는 매월 발행본의 디지털 메타데이터를 관리한다.

- 발행월
- 호수명
- 표지 이미지
- PDF 또는 웹진 링크
- 공개 여부

이는 `basic` 또는 별도 공개 화면에서 갤러리 형태로 소비될 수 있다.

### 10.4. 도메인 규칙

- 발송 횟수는 수동 편집보다 발송 처리 이벤트 기준으로 증가시키는 것이 원칙이다.
- 현재 활성 구독의 배송지는 스냅샷 수정 방식으로 관리한다.
- 과거 발송 이력과 현재 구독 상태를 혼동하지 않도록 이력을 분리한다.

## 11. 모듈 간 관계와 데이터 소유권

### 11.1. 소유권 원칙

- 회원 인증과 주소록 마스터는 `member`가 소유한다.
- 그룹/모임/공지의 쓰기 책임은 `service`가 소유한다.
- 공개 조회 조합은 `basic`이 소유한다.
- 주문과 재고는 `store`가 소유한다.
- 구독과 발송 이력은 `heart`가 소유한다.

### 11.2. 참조 방식

- `basic`은 `service`가 관리하는 공개 가능한 그룹/모임/공지 데이터를 조회 모델로 소비한다.
- `store`와 `heart`는 `memberId`만 저장하고, 거래 시점 데이터는 자체 스냅샷으로 보존한다.
- `service`와 `member`의 연결도 가능하면 ID와 권한 서비스 단위로 유지한다.

### 11.3. 금지 규칙

- 한 모듈의 영속 엔티티를 다른 모듈에서 임의 수정하지 않는다.
- 외부 모듈 테이블에 직접 접근하는 쿼리를 무분별하게 작성하지 않는다.
- 조회 편의를 이유로 강한 양방향 연관관계를 만들지 않는다.

## 12. 대표 유스케이스

### 12.1. 일반 사용자의 모임 검색

1. 사용자가 지역/요일/모임 성격을 선택한다.
2. `basic`이 공개 가능한 모임 목록을 시간순으로 조회한다.
3. 결과에는 그룹 기본 정보와 장소/공지 요약이 함께 제공될 수 있다.

### 12.2. 관리자의 그룹 정보 수정

1. 관리자가 그룹 정보를 수정한다.
2. `service`가 권한과 입력값을 검증한다.
3. 그룹을 수정하고 `GroupChangeLog`를 남긴다.
4. 이후 `basic` 조회 결과에 최신 정보가 반영된다.

### 12.3. 회원의 도서 주문

1. 회원이 주소록에서 배송지를 선택한다.
2. `store`가 결제 직전 배송지 원본과 상품 가격을 읽는다.
3. 결제 완료 시 `Order`, `OrderItem` 스냅샷을 생성한다.
4. 관리자가 배송 라벨 추출 후 발송 처리한다.

### 12.4. 월간지 구독 발송

1. 관리자가 해당 월 발송 대상을 조회한다.
2. 엑셀 추출 또는 발송 배치를 실행한다.
3. 각 대상마다 `DeliveryHistory`를 생성한다.
4. `sentCount`를 증가시키고 필요 시 `EXPIRED`로 전환한다.

## 13. 상태와 스냅샷 설계 원칙

### 13.1. 현재 상태와 과거 사실의 분리

다음 두 가지는 반드시 구분한다.

- 현재 유효한 운영 상태
- 과거 특정 시점의 사실 기록

예시:

- 회원 현재 주소와 주문 당시 배송지는 다르다.
- 현재 그룹 정보와 변경 이력의 과거 값은 다르다.
- 현재 구독 수령지와 지난달 발송 이력은 다르다.

### 13.2. 스냅샷이 필요한 대표 사례

- 주문 배송지
- 주문 품목 가격
- 구독 수령인 정보
- 변경 전/후 그룹 데이터

### 13.3. 스냅샷 저장 형식

텍스트 컬럼 조합 또는 JSON 컬럼 중 하나를 사용하되, 다음 기준을 만족해야 한다.

- 사람이 확인 가능해야 한다
- 운영 엑셀 추출이 쉬워야 한다
- 과거 값이 원본 변경에 영향받지 않아야 한다

## 14. 구현 가이드라인

### 14.1. 패키지/코드 구조

- 모듈 루트 아래에 API, application, domain, infrastructure를 둘 수 있다.
- 단, 기술 계층보다 도메인 경계가 상위 개념이어야 한다.
- 외부 공개 API DTO와 내부 도메인 객체를 분리한다.

예시:

```text
modules/store/order/
  api/
  application/
  domain/
  infrastructure/
```

### 14.2. API 설계

- 조회 API와 명령 API를 역할에 맞게 분리한다.
- 관리자 API는 `/admin` 또는 별도 네임스페이스를 고려한다.
- 공개 API는 캐시 친화적이고 조합된 응답을 우선한다.

### 14.3. DB 설계

- 트랜잭션 경계는 가능한 한 모듈 내부에서 닫는다.
- 모듈 간 FK는 최소화하고, 필요한 경우에도 물리 FK보다 애플리케이션 규칙 우선을 검토한다.
- 조회 성능이 중요한 화면은 인덱스와 projection을 우선 설계한다.

### 14.4. 테스트 우선순위

우선적으로 보호해야 할 테스트는 다음과 같다.

- 모임 검색 필터링/정렬 규칙
- 그룹 변경 이력 생성
- 주문 스냅샷 고정
- 구독 발송 횟수 증가와 만료 전환
- 권한별 접근 제어

## 17. 최종 설계 원칙 요약

이 프로젝트는 다음 다섯 가지 원칙을 핵심으로 유지한다.

1. 도메인별로 경계를 나눈 모듈러 모놀리스로 시작한다.
2. 공개 조회는 `basic`, 운영 쓰기는 `service`로 분리하는 CQRS-Lite를 적용한다.
3. `member`는 공통 마스터이지만, 거래 도메인은 식별자 참조와 스냅샷으로 결합도를 낮춘다.
4. 주문, 구독, 변경 이력은 과거 사실을 보존하는 스냅샷 패턴을 따른다.
5. 현재 상태와 과거 기록을 분리해 운영, 감사, 서비스 분리에 모두 대비한다.
