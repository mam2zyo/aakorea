# A.A. KOREA 개발 로드맵

기준일: 2026-03-25

이 문서는 `README.md`, `docs/REQUIREMENTS.md`, `docs/DOMAIN_SPEC.md`, 그리고 현재 저장소 구현 상태를 함께 대조해 정리한 권장 개발 순서다.
기간은 별도 인력 정보가 없으므로 "1개 풀스택 스쿼드 기준 러프 추정"으로 본다.

## 1. 우선순위 판단 기준

이번 로드맵은 아래 원칙으로 우선순위를 정했다.

1. 초심자와 비로그인 방문자가 모임을 빠르게 찾고 불안을 줄일 수 있는 기능을 최우선으로 둔다.
2. 문서의 `Must Have`를 먼저 충족하되, 선행 의존성이 큰 공통 기반(`member`, 인증/인가)은 도메인 기능보다 앞선다.
3. 공개 정보의 최신성과 운영 안전성을 위해 `service`의 권한, 공지, 변경 이력 완성을 거래 기능보다 먼저 처리한다.
4. `store`, `heart`는 모두 중요하지만, `store`가 주소록/스냅샷 패턴을 먼저 검증하기에 더 적은 위험으로 시작할 수 있다.
5. 테스트 우선순위는 도메인 명세서의 보호 대상 규칙을 그대로 따른다.

## 2. 현재 상태 요약

| 영역 | 문서상 기대 상태 | 현재 구현 상태 | 판단 |
| --- | --- | --- | --- |
| `basic` | 모임 검색, 그룹 상세, 공지 통합, 초심자 콘텐츠, GIS 검색 | 모임 검색(지역/요일), 그룹 상세, 공지 조회 일부 구현 | 핵심 조회 뼈대는 있음. 초심자 콘텐츠와 검색 고도화가 비어 있음 |
| `service` | 지역연합, GSR, 그룹, 모임, 공지, 변경 이력, 권한 운영 | 지역연합/그룹/GSR CRUD와 그룹별 모임 CRUD 구현 | 공지 관리, 인증/인가, 권한 제어, GSR 셀프서비스가 미완 |
| `member` | JWT 로그인, 역할, 프로필, 주소록 | 패키지 구조만 존재 | 실질 구현 전 |
| `store` | 상품, 장바구니, 주문, 재고, 스냅샷, 배송 운영 | 패키지 구조만 존재 | 실질 구현 전 |
| `heart` | 구독, 발송 이력, 만료, 선물, 디지털 아카이브 | 패키지 구조만 존재 | 실질 구현 전 |

추가로 현재 백엔드에는 `spring-security` 또는 JWT 관련 구현이 아직 없어 `/admin` 계열 기능이 실질적으로 보호되지 않는 상태로 보는 것이 맞다.

## 3. 권장 개발 순서

### Phase 0. 기반 안정화

권장 기간: 1주

목표:
- 현재 구현된 `basic` + `service` 기능을 이후 확장에 버틸 수 있는 상태로 정리한다.
- 인증/인가, 거래 도메인 추가 전에 모듈 경계와 테스트 기준을 고정한다.

주요 작업:
- `basic` 조회 API와 `service` 관리 API의 네임스페이스/DTO 규칙 확정
- 공통 예외 응답, 입력 검증, 시드 데이터 전략 정리
- 테스트 우선순위에 맞춘 기본 테스트 템플릿 정비
- DB 마이그레이션 도입 여부 결정

완료 기준:
- 모임 검색/그룹 상세/관리자 CRUD가 로컬에서 일관되게 동작
- 이후 단계에서 재사용할 테스트/시드/예외 처리 기준 문서화

### Phase 1. 공개 사용자 MVP 완성

권장 기간: 2주

왜 먼저인가:
- 제품 목표의 1순위는 초심자와 비로그인 방문자가 A.A. 정보와 모임에 빠르게 연결되는 것이다.
- 현재 저장소도 이 흐름이 가장 앞서 구현되어 있어, 가장 빠르게 사용자 가치를 완성할 수 있다.

주요 작업:
- `처음 오신 분들께`, `A.A. 소개`, 첫 참석 가이드, FAQ/자가진단 등 공개 콘텐츠 화면 추가
- 모임 검색에 `MeetingType` 필터 추가
- 검색 결과에서 공지 노출 규칙 보강
- 그룹 상세 화면의 정보 구성 정리
- 공개 정보 메뉴 구조 정리

완료 기준:
- 로그인 없이 "안내 콘텐츠 확인 -> 모임 검색 -> 그룹 상세 확인"이 한 흐름으로 이어짐
- 문서의 `Must Have` 중 공개 영역 항목 충족

테스트 게이트:
- 모임 검색 필터링/정렬 규칙
- 공지 노출 조건

### Phase 2. 인증/권한 + 운영 백오피스 완성

권장 기간: 2~3주

왜 여기인가:
- 문서상 `JWT 로그인과 역할 기반 접근 제어`는 `Must Have`다.
- 관리자/GSR 기능이 보호되지 않은 채 `store`, `heart`를 올리면 운영 리스크가 너무 크다.

주요 작업:
- `member` 최소 구현: `Member`, 역할, 상태, 로그인
- Spring Security + JWT 기반 인증/인가 도입
- 관리자 API 및 프런트 라우트 보호
- GSR 승인/정지/그룹 매핑 정책 구현
- 그룹 공지 CRUD 추가
- 그룹 변경 이력 생성/조회 보강
- GSR가 자기 그룹만 수정할 수 있는 셀프서비스 MVP 제공

완료 기준:
- `ADMIN`, `GSR`, 일반 사용자의 접근 범위가 명확히 분리됨
- 운영자가 그룹/모임/공지 변경을 안전하게 처리 가능
- 변경 내역이 감사 가능하게 남음

테스트 게이트:
- 권한별 접근 제어
- 그룹 변경 이력 생성
- GSR 본인 그룹 한정 수정 규칙

### Phase 3. 멤버 공간과 주소록 기반 완성

권장 기간: 2주

왜 별도 단계인가:
- `store`, `heart` 둘 다 주소록과 사용자 식별자에 의존한다.
- 문서상 두 거래 도메인은 `member`를 직접 참조하지 않고 식별자 + 스냅샷으로 연결해야 하므로, 중간 기반 단계가 필요하다.

주요 작업:
- 회원 프로필 조회/수정
- 배송지 CRUD
- 기본 배송지 지정
- `store`, `heart`가 사용할 주소록 조회 DTO 설계
- 마이페이지 또는 최소 계정 화면 추가

완료 기준:
- 로그인한 사용자가 여러 배송지를 등록하고 기본 배송지를 지정 가능
- 주문/구독 직전 선택 가능한 주소 원본이 준비됨

### Phase 4. Store MVP

권장 기간: 3주

왜 `heart`보다 먼저인가:
- 주소록 스냅샷, 가격 스냅샷, 상태 전이 같은 핵심 패턴을 더 단순한 거래 흐름에서 먼저 검증할 수 있다.
- `heart`보다 상태 조합이 적어 위험이 낮다.

주요 작업:
- 상품/재고 관리
- 상품 목록/상세
- 장바구니
- 주문 생성과 결제 시점 스냅샷 저장
- 주문 상태(`PENDING_PAYMENT`, `PAID`, `SHIPPED`, `CANCELLED`) 관리
- 관리자 오프라인 주문 등록
- 사용자 주문 내역 조회

완료 기준:
- 사용자가 배송지를 선택해 주문을 생성 가능
- 주문 생성 후 주소/가격 변경이 과거 주문에 영향을 주지 않음
- 운영자가 결제 완료 주문을 발송 상태로 관리 가능

테스트 게이트:
- 주문 스냅샷 고정
- 재고 차감/복구 규칙

### Phase 5. Heart MVP

권장 기간: 3주

주요 작업:
- 구독 플랜 관리
- 장기 구독 생성과 연장
- 선물 구독
- 수령인 스냅샷 저장
- `DeliveryHistory` 생성
- `sentCount` 증가와 `EXPIRED` 전환
- 관리자 발송 대상 조회
- 발행본(`MagazineIssue`) 기본 관리

완료 기준:
- 관리자가 월간 발송 명단을 만들고 발송 이력을 남길 수 있음
- 사용자가 활성 구독, 남은 횟수, 만료 상태를 확인 가능
- 선물 구독과 비회원 수령인 케이스를 처리 가능

테스트 게이트:
- 구독 발송 횟수 증가와 만료 전환
- 활성 구독 배송지 변경과 과거 발송 이력 분리

### Phase 6. 운영 자동화와 고도화

권장 기간: 2~3주

이 단계에서 다룰 항목:
- 지도 기반 주변 검색
- 주문/발송 엑셀 추출
- 만료 임박 자동 알림
- 디지털 아카이브 공개 화면
- 모바일 운영 UX 개선
- 운영 대시보드
- 통합 검색/PWA/다국어 같은 확장 과제

이 단계로 미루는 이유:
- 문서 기준으로 대부분 `Should Have` 또는 `Could Have`다.
- 앞 단계의 인증, 거래, 운영 정합성이 먼저 안정화되어야 효과가 크다.

1. Spring Security/JWT 도입 설계
2. `Member`/역할/상태 최소 모델 정의
3. 그룹 공지 CRUD API와 관리자 화면
4. 그룹 변경 이력 생성 규칙 보강
5. 공개용 `처음 오신 분들께` 콘텐츠 화면
6. 모임 검색 `MeetingType` 필터 추가
7. 주소록 엔티티와 API 설계
8. `store` 주문 스냅샷 모델 설계
9. `heart` 구독/발송 이력 상태 모델 설계
10. 테스트 우선순위 기준으로 회귀 테스트 보강

## 6. 후순위로 미뤄도 되는 항목

- PWA 전환
- 운영 대시보드
- 통합 검색 고도화
- 다국어 지원
- 지도 공급자 최종 최적화
- 완전한 CMS 수준의 콘텐츠 관리

이 항목들은 가치가 없어서가 아니라, 현재 문서 기준 `Must Have` 충족과 공통 기반 완성보다 선행할 이유가 약하다.
