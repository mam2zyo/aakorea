# 도메인별 관리자 화면 분리 검토

작성일: 2026-03-26

## 1. 검토 질문

다음 방향이 타당한지 검토한다.

- `service` 도메인은 조직, 그룹, 모임, 공지, 공개 콘텐츠 관리까지만 담당한다.
- `store` 도메인의 상품 입력, 재고 관리, 주문 처리 등은 `service`에 몰아넣지 않고 별도 운영 화면으로 분리한다.
- `heart` 도메인의 구독, 발송, 회차 관리 등도 `service`에 몰아넣지 않고 별도 운영 화면으로 분리한다.

이 문서는 기존 문서인 `DOMAIN_SPEC`, `REQUIREMENTS`, `ROADMAP`, 그리고 프론트엔드 구조 검토 문서를 바탕으로 위 방향의 타당성을 예측한다.

## 2. 먼저 정리해야 하는 핵심 전제

이 논점에서 가장 중요한 것은 아래 두 개를 구분하는 것이다.

- `관리자(Admin)`는 사용자 역할이다.
- `service`, `store`, `heart`는 비즈니스 도메인이다.

즉 "관리자가 쓰는 화면"이라는 이유만으로 그 화면이 모두 `service` 도메인에 속해야 하는 것은 아니다.

이 전제를 놓치면 아래 같은 혼선이 생긴다.

- 조직 관리도 `service`
- 상품/재고도 `service`
- 구독/발송도 `service`

결국 `service`가 "운영자 화면을 다 모아둔 폴더"가 되어 버리고, 도메인 문서에서 정의한 책임 분리가 프론트엔드에서 무너진다.

## 3. 문서 기준으로 본 현재 도메인 책임

기존 문서는 이미 도메인 책임을 비교적 명확하게 나눠 두고 있다.

### 3.1. `service`

문서상 `service`는 아래 책임을 가진다.

- 조직(`District`) 관리
- GSR 및 권한 관련 운영
- 그룹/모임 관리
- 공지 및 변경 이력 관리
- 공개 조회(`basic`)에 반영되는 운영성 데이터의 command 책임

즉 `service`는 A.A. 조직 운영과 공개 정보 최신성 유지에 가까운 도메인이다.

### 3.2. `store`

문서상 `store`는 아래 책임을 가진다.

- 상품
- 장바구니
- 주문
- 재고
- 오프라인 주문
- 주문 스냅샷과 배송 처리

즉 `store`는 전형적인 거래 도메인이다.

### 3.3. `heart`

문서상 `heart`는 아래 책임을 가진다.

- 정기 구독
- 선물 구독
- 발송 이력
- 만료 처리
- 발행본(`MagazineIssue`) 관리

즉 `heart`는 정기 구독과 발송 수명주기를 다루는 도메인이다.

### 3.4. 문서가 이미 암시하는 방향

기존 문서에는 다음과 같은 강한 힌트가 있다.

- `store.aakorea.org`로 분리 가능한 구조를 목표로 둔다.
- `heart.aakorea.org`로 분리 가능한 구조를 목표로 둔다.
- 주문/재고는 `store`가 소유한다.
- 구독/발송 이력은 `heart`가 소유한다.
- 그룹/모임/공지 쓰기 책임은 `service`가 소유한다.

이 관점에서 보면 운영 화면도 같은 기준을 따르는 편이 자연스럽다.

## 4. 판단

### 결론

질문하신 분리 방향은 타당할 뿐 아니라, 현재 문서 구조를 일관되게 유지하려면 오히려 그렇게 가는 편이 더 맞다.

더 구체적으로 말하면 다음이 적절하다.

- `service` 운영 화면은 조직/그룹/모임/공지/공개 콘텐츠 관리에 집중한다.
- `store` 운영 화면은 상품/재고/주문/배송 처리에 집중한다.
- `heart` 운영 화면은 구독/발송/회차/아카이브 관리에 집중한다.

즉 "운영 화면을 도메인별로 나눈다"는 방향이 문서상 책임 분리와 더 잘 맞는다.

## 5. 왜 이 분리가 타당한가

### 5.1. 도메인 소유권과 UI 소유권이 일치한다

백엔드 문서에서 주문과 재고는 `store`, 구독과 발송은 `heart`, 그룹과 공지는 `service`가 소유한다.

프론트엔드 운영 화면도 같은 기준을 따르면 아래 장점이 생긴다.

- 어느 화면이 어느 도메인의 규칙을 따르는지 명확해진다.
- 도메인별 테스트와 변경 범위를 좁히기 쉽다.
- "이 기능은 왜 service에 있지?" 같은 혼란이 줄어든다.

### 5.2. 업무 흐름이 서로 너무 다르다

`service`, `store`, `heart`는 운영자가 본다고 해서 비슷한 화면이 아니다.

- `service`: 조직 관리, 목록 편집, 공지, 변경 이력
- `store`: 상품 카탈로그, 재고 수량, 주문 상태, 배송 처리
- `heart`: 구독 수명주기, 회차, 발송 배치, 만료 처리

이 흐름들을 한 도메인 안에 모으면 공통화보다 예외 처리가 더 빨리 늘어난다.

### 5.3. 권한 분리에도 유리하다

장기적으로는 관리자 역할도 단일 `ADMIN` 하나로 끝나지 않을 가능성이 높다.

예시:

- `SERVICE_ADMIN`
- `STORE_ADMIN`
- `HEART_ADMIN`
- `SUPER_ADMIN`

이런 역할 분리는 도메인별 운영 화면이 분리되어 있을 때 훨씬 자연스럽다. 반대로 모든 운영 화면이 `service` 아래에 모이면 권한 경계도 느슨해지기 쉽다.

### 5.4. 배포 분리 가능성과도 맞는다

문서가 이미 `store.aakorea.org`, `heart.aakorea.org` 같은 분리 가능성을 열어 두고 있기 때문에, 프론트엔드도 처음부터 물리 분리까지 염두에 둔 논리 분리를 해 두는 편이 좋다.

중요한 점은 지금 당장 별도 리포지토리나 별도 배포를 강제하자는 뜻은 아니다.

다만 아래는 미리 분리하는 편이 좋다.

- 라우트 그룹
- 화면 shell
- 상태와 API 모듈
- 운영 메뉴 구조

## 6. 다만 조심해야 할 점

### 6.1. `admin` 네임스페이스와 `service` 도메인을 혼동하지 말아야 한다

여기서 가장 좋은 구분은 다음이다.

- `admin`은 접근 표면(surface) 또는 역할 기반 입구
- `service`, `store`, `heart`는 그 안에서 실제 기능을 소유하는 도메인

즉 아래처럼 생각하면 된다.

- `/admin/service/*`
- `/admin/store/*`
- `/admin/heart/*`

이 구조는 "한 관리자 포털 안에 여러 도메인 운영 영역이 있다"는 뜻이지, `store`와 `heart`가 `service`의 하위 도메인이라는 뜻이 아니다.

### 6.2. 인증과 공통 UI까지 완전히 찢을 필요는 없다

도메인별 운영 화면은 분리하되, 아래는 공용으로 유지하는 편이 좋다.

- 로그인과 세션
- 공통 권한 체크
- 공통 디자인 시스템
- 공통 테이블/폼 UI 컴포넌트
- 공통 fetch client

즉 "도메인별 운영 화면 분리"는 필요하지만, "모든 걸 별도 앱으로 즉시 해체"까지는 아니다.

### 6.3. 통합 운영 대시보드는 별도 계층으로 두는 것이 낫다

만약 관리자가 전체 현황을 한눈에 보는 홈이 필요하다면, 그 화면은 `service`가 아니라 중립적인 운영 포털 계층에 두는 것이 낫다.

예를 들면 아래 같은 구조다.

- `admin home`: 전체 운영 포털 홈
- `service admin`: 조직/콘텐츠 운영
- `store admin`: 상품/주문 운영
- `heart admin`: 구독/발송 운영

이렇게 해야 "통합 대시보드가 있으니 모든 기능도 service에 넣는다"는 식의 역전이 생기지 않는다.

## 7. 문서들 사이의 해석 차이

기존 `REQUIREMENTS`에는 관리자 요구사항이 비교적 넓게 적혀 있어서, 읽는 방식에 따라 "관리자는 service 도메인으로 모든 운영을 한다"처럼 보일 수 있다.

하지만 이 부분은 아래처럼 해석하는 편이 더 일관된다.

- `관리자`는 여러 도메인을 사용하는 사용자 그룹이다.
- `service`는 그중 조직/공개정보 운영을 담당하는 하나의 도메인이다.
- `store`, `heart` 관련 운영 업무는 각각의 도메인 운영 기능으로 본다.

즉 문서에서 충돌이 있다기보다, "사용자 역할"과 "도메인 경계"가 같은 문장 안에 같이 적혀 있어서 혼동이 생긴 것으로 보는 편이 맞다.

## 8. 프론트엔드 구조에 대한 권장 해석

앞으로의 프론트엔드 구조는 아래처럼 가는 것이 좋다.

```text
src/
  app/
    router/
    providers/
  shared/
  domains/
    basic/
    member/
    service/
      public-content-admin/
      districts/
      groups/
      meetings/
      notices/
      gsrs/
    store/
      shop/
      admin/
        items/
        inventory/
        orders/
    heart/
      subscription/
      admin/
        plans/
        subscriptions/
        deliveries/
        issues/
```

핵심은 아래와 같다.

- `service` 안에 `store admin`, `heart admin`을 넣지 않는다.
- `store`와 `heart`는 각 도메인 안에서 사용자용 화면과 관리자용 화면을 함께 가질 수 있다.
- 최상위 라우터는 이들을 조립만 하고, 소유권은 각 도메인에 둔다.

## 9. 추천하는 운영 화면 구조

단기적으로 가장 현실적인 구조는 다음이다.

### 옵션 A. 하나의 관리자 포털, 세 개의 도메인 영역

추천도: 가장 높음

예시:

- `/admin`
- `/admin/service/*`
- `/admin/store/*`
- `/admin/heart/*`

장점:

- 로그인과 네비게이션을 공유할 수 있다.
- 운영자는 한 포털 안에서 이동할 수 있다.
- 도메인 경계는 유지된다.
- 향후 물리 분리 전 단계로 적합하다.

주의점:

- `/admin`은 중립 namespace여야 한다.
- `service`가 상위 개념처럼 보이지 않게 메뉴/코드 구조를 설계해야 한다.

### 옵션 B. 도메인별 별도 운영 앱

추천도: 중장기 선택지

예시:

- `ops.aakorea.org/service`
- `store.aakorea.org/admin`
- `heart.aakorea.org/admin`

장점:

- 배포와 접근 제어를 더 강하게 분리할 수 있다.
- 도메인 독립성이 가장 높다.

주의점:

- 로그인, 공통 UI, 이동 경험이 분절될 수 있다.
- 지금 단계에서는 운영 부담이 클 수 있다.

## 10. 권장 결론

현재 시점에서 가장 균형 잡힌 방향은 아래다.

### 권장안

- `service`는 조직/그룹/모임/공지/공개 콘텐츠 운영 도메인으로 한정한다.
- `store` 운영 기능은 `store` 도메인 안의 별도 관리자 영역으로 둔다.
- `heart` 운영 기능은 `heart` 도메인 안의 별도 관리자 영역으로 둔다.
- 프론트엔드 입구는 필요하면 `/admin`으로 공유하되, 그 내부를 `service/store/heart`로 명확히 분리한다.
- 인증과 공통 UI는 공유하되, 기능 소유권은 공유하지 않는다.

이 방향은 다음을 동시에 만족한다.

- 기존 도메인 문서와의 일관성
- 프론트엔드 확장성
- 권한 분리 가능성
- 향후 서브도메인/별도 앱 분리 가능성

## 11. 한 줄 요약

`service`에 모든 관리자 화면을 몰아넣기보다, `관리자`는 역할로 두고 실제 운영 화면은 `service`, `store`, `heart`의 도메인 경계에 맞춰 분리하는 쪽이 더 타당하다.
