# A.A. KOREA 웹앱 도메인 명세서: Basic + Service

공통 아키텍처 원칙과 모듈 간 공통 규칙은 [DOMAIN_SPEC.md](./DOMAIN_SPEC.md)를 따른다.
이 문서는 공개 조회를 담당하는 `basic`과 운영 쓰기를 담당하는 `service`의 책임 경계와 연동 흐름만 정리한다.

## 1. 책임 경계

| 도메인 | 소유 책임 | 주요 사용자 |
| ------ | --------- | ----------- |
| `basic` | 공개 조회, 모임 검색, 그룹 상세, 행사/자료/문헌 안내 소비 | 방문자, 초심자, 멤버 |
| `service` | 조직, 그룹, 모임, 공지, 행사, 공개 콘텐츠의 생성과 수정 | ADMIN, GSR |

핵심 원칙은 다음 한 줄로 요약된다.

- `service`가 원천 데이터를 관리하고, `basic`이 공개 가능한 상태만 읽기 모델로 조합한다.

---

## 2. `basic` 도메인 고유 내용

### 2.1. 핵심 읽기 모델

- `Group`
- `Meeting`
- `MeetingPlace`
- `GroupNotice`
- `District` 공개 디렉터리용 요약 모델
- `Event`
- `PublicationCatalog`
- `PublicResource`

### 2.2. 핵심 기능

- `Province`, `DayOfWeek` 기반 모임 검색
- 시간순 정렬된 공개 모임 목록 제공
- 그룹 기본 정보, 정기 모임 일정, 활성 공지, 연락처, 장소를 묶은 그룹 상세 조회
- 초심자, 가족, 전문가 대상 정적, 반정적 공개 콘텐츠 제공
- 지역연합 소개, 공식 링크 모음, 역사/연혁, 팩트 파일 같은 공개 자료 제공
- 공개 행사 목록/상세와 포스터형 부가 자료 제공
- 주요 도서/소책자 소개 카탈로그를 제공하고 필요 시 `store`로 연결
- 공개 콘텐츠는 최소한 다음 범주를 포함할 수 있다.
  - A.A. 소개와 기본 원칙
  - 자기 점검형 안내(`A.A.는 당신을 위한 것인가?`)
  - 새로 나온 사람 FAQ, 첫 방문 안내
  - 젊은이, 여성, 전문가 대상 안내
  - 익명성 원칙, 장문 FAQ(`44가지 질문들`)
  - 지역연합, 사이트 모음, 역사/아카이브
  - 팩트 파일, 공개 문헌/리플릿, 문헌 소개 카탈로그
  - 행사 안내, 온라인 모임 변경 안내, 일반 공지

### 2.3. 추가 요구

- `MeetingPlace`는 위도, 경도 좌표를 가질 수 있다.
- PostgreSQL + PostGIS 기반 주변 검색과 지도 시각화를 고려한다.
- 복잡한 관리자 내부 상태 대신 현재 공개 상태만 노출한다.

## 3. `service` 도메인 고유 내용

### 3.1. 핵심 엔티티

- `District`
- `Group` (Command 모델)
- `Meeting` (Command 모델)
- `GSR`
- `GroupChangeLog`
- `Event`
- 공지 및 공개 콘텐츠 관리 모델

### 3.2. 핵심 기능

- 지역연합 생성, 수정, 삭제와 그룹 매핑 관리
- GSR 계정 생성, 승인, 정지, 담당 그룹 연결
- 그룹 기본 정보와 정기 모임의 추가, 수정, 삭제
- 임시 휴무, 장소 변경, 행사 안내 같은 공지 운영
- 행사 등록, 수정, 게시와 행사 포스터/연락처 관리
- 소개, 안내, 소식형 공개 콘텐츠 작성과 게시
- 지역연합 소개, 외부 링크 모음, 역사/연혁, 팩트 파일 같은 공개 자료 관리
- 문헌/소책자 소개 카탈로그 작성과 게시
- 공개 콘텐츠는 대상별 분류와 게시 상태를 가져야 한다.
  - 예: 초심자, 가족/지인, 전문가, 일반 소개, 정책/원칙

### 3.3. 운영 규칙

- 공개 노출에 영향을 주는 변경은 `GroupChangeLog` 등 감사 가능한 방식으로 기록한다.
- 공지의 공개 여부와 노출 기간은 `basic` 조회 결과에 직접 반영된다.
- 입력 검증과 상태 전이는 `service`에서 강하게 통제한다.
- 행사는 정기 모임(`Meeting`)이나 일반 공지(`GroupNotice`)와 구분되는 별도 공개 리소스로 관리한다.
- `members.html` 계열의 지역연합, 사이트 모음, 역사, 온라인 모임 변경 같은 공개 정보는 `member`가 아니라 `basic/service` 공개 정보 축으로 본다.
- `books.html` 계열의 문헌 소개/카탈로그는 `basic/service` 공개 정보 축으로 보고, 가격/재고/주문은 `store`가 소유한다.

## 4. `basic`과 `service`의 연동 규칙

- 동일한 개념이라도 읽기 모델과 쓰기 모델은 API, DTO, 유스케이스를 분리한다.
- `basic`은 `service`의 관리자 내부 상태를 그대로 노출하지 않는다.
- `service`는 운영 계정과 `memberId`를 연결해 권한을 판정한다.
- 그룹, 모임, 공지의 최종 쓰기 책임은 `service`가 가진다.
- 공개 조회 조합의 최종 책임은 `basic`이 가진다.
- `basic`의 공개 문헌 소개 페이지는 `store`로 이동하는 링크/CTA를 포함할 수 있지만, 판매 가능 여부와 거래 처리는 `store` 책임으로 유지한다.
- 온라인 모임 시간 변경 같은 운영성 안내는 행사로 흡수하지 않고, 모임/공지 읽기 모델과 연동해 노출한다.

대표 흐름은 다음과 같다.

1. 운영자가 `service`에서 그룹, 모임, 공지, 콘텐츠를 수정한다.
2. 공개 가능한 상태와 노출 기간이 확정된다.
3. `basic`이 이를 읽기 모델로 조합해 검색 결과와 상세 페이지에 반영한다.

---

## 5. 구현 초점

- 모임 검색 필터링과 정렬 규칙
- 그룹 상세 통합 조회와 활성 공지 반영
- 행사와 공지, 문헌 소개의 정보 구조 분리
- 지역연합 디렉터리, 링크 모음, 역사 자료 같은 공개 허브 재구성
- 그룹 변경 이력 생성
- 권한별 접근 제어
- 공개 콘텐츠 게시 상태와 노출 제어

---

## 6. 확장 포인트

### 6.1. `basic`

- 지도 기반 주변 검색 고도화
- 공지, 행사, 문헌 공개 콘텐츠 통합 검색
- 다국어 또는 접근성 강화

### 6.2. `service`

- GSR 셀프서비스 편집 기능
- 승인 워크플로우
- 운영 통계 대시보드
