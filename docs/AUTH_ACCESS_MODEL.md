# 인증 및 접근 제어 모델

작성일: 2026-03-26

## 1. 목적

이 문서는 A.A. KOREA 웹앱의 인증, 계정 상태, 접근 제어 모델을 구현 전에 고정하기 위한 기준 문서다.
공통 도메인 원칙은 `DOMAIN_SPEC.md`, 사용자 요구는 `REQUIREMENTS.md`를 따른다.

본 문서는 다음 질문에 답한다.

- 사용자는 어떻게 로그인하는가
- 로그인 이후 어떤 단계를 거쳐 권한을 판정하는가
- 앱별 관리자 표면과 사용자 화면은 어떤 기준으로 보호하는가
- `ADMIN`, `GSR`, 앱 도메인 운영 권한은 어떻게 분리하는가

---

## 2. 인증과 인가를 분리해서 본다

프로젝트에서는 다음 세 층을 분리한다.

1. 인증(Authentication): 사용자가 누구인지 확인한다.
2. 계정 상태(Account Status): 로그인 가능한 상태인지 확인한다.
3. 인가(Authorization): 특정 유스케이스를 실행할 권한이 있는지 확인한다.

즉 "로그인했다"는 사실만으로 그룹 수정, 주문 관리, 발송 관리 권한을 자동으로 주지 않는다.

---

## 3. 로그인 모델

### 3.1. 기본 원칙

- 외부 IdP 연동은 `OIDC` 또는 `OAuth2`를 허용한다.
- 로그인 성공 후에는 내부 세션 또는 JWT 기반 토큰을 발급한다.
- 프런트엔드는 내부 토큰 기준으로 인증 상태를 판단한다.

### 3.2. 최소 인증 주체

로그인 후 애플리케이션이 알아야 하는 최소 정보는 다음과 같다.

- `userId`
- display name 또는 식별 가능한 기본 이름
- 계정 상태
- 전역 역할 집합

### 3.3. 토큰에 넣을 최소 클레임

- `sub`: 내부 사용자 식별자 또는 외부 subject
- `userId`
- `roles`
- `status`
- `issuedAt`, `expiresAt`

도메인별 세부 권한 목록 전체를 토큰에 과도하게 넣기보다, 필요 시 서버에서 재판정하는 방식을 기본으로 한다.

---

## 4. 계정 상태 모델

최소 제안 상태는 다음과 같다.

- `ACTIVE`: 정상 사용 가능
- `PENDING_APPROVAL`: 승인 대기, 일부 보호 기능 제한
- `SUSPENDED`: 로그인 또는 주요 기능 제한
- `WITHDRAWN`: 탈퇴 또는 비활성

세부 명칭은 구현 시 조정 가능하지만, 아래 규칙은 유지한다.

- 인증 성공과 기능 사용 가능 여부는 같은 판정이 아니다.
- `GSR` 또는 운영 계정은 역할뿐 아니라 상태도 함께 봐야 한다.

---

## 5. 역할과 도메인 권한 모델

### 5.1. 전역 역할

전역 역할은 작게 유지한다.

- `USER`
- `GSR`
- `ADMIN`

### 5.2. 도메인별 운영 권한

다음 권한은 전역 역할만으로 모두 표현하지 않는다.

- `general.admin(service)` 운영 권한
- `store` 운영 권한
- `heart` 운영 권한

권장 방식은 다음과 같다.

- 전역 역할은 인증과 대분류 접근 제어에 사용한다.
- 앱 도메인별 운영 자격은 각 도메인의 정책과 할당 정보로 관리한다.

예시:

- `ADMIN`이더라도 `store` 운영 화면 접근 여부는 별도 정책으로 제한할 수 있다.
- `GSR`이더라도 자신에게 매핑된 그룹만 수정할 수 있다.

---

## 6. 접근 제어 판정 순서

모든 보호 기능은 아래 순서로 판정한다.

1. 로그인 여부 확인
2. 계정 상태 확인
3. 전역 역할 확인
4. 도메인별 정책 확인
5. 자원 소유권 또는 범위 확인

예시:

- 그룹 수정: 로그인 -> 활성 계정 -> `GSR` 또는 `ADMIN` -> 담당 그룹 매핑 확인
- 주문 관리: 로그인 -> 활성 계정 -> 운영자 권한 -> `store` 관리자 정책 확인
- 발송 처리: 로그인 -> 활성 계정 -> 운영자 권한 -> `heart` 관리자 정책 확인

---

## 7. 표면별 접근 정책

| 표면 | 인증 필요 | 기본 접근 기준 |
| ---- | --------- | -------------- |
| 공개 화면 (`general.public`, 별칭 `basic`) | 아니오 | 누구나 접근 가능 |
| 내 공간 (`user`) | 예 | 활성 계정 |
| `/admin/service/*` (`general.admin`) | 예 | `ADMIN` 또는 적절한 `GSR` + `general.admin(service)` 정책 |
| `/admin/store/*` (`store.admin`) | 예 | 운영자 + `store` 정책 |
| `/admin/heart/*` (`heart.admin`) | 예 | 운영자 + `heart` 정책 |

`admin`은 접근 표면이지, 도메인 소유권을 의미하지 않는다.

---

## 8. 프런트엔드/백엔드 책임 분리

### 8.1. 프런트엔드

- 로그인 상태와 계정 상태를 기반으로 라우트 접근을 1차 제어한다.
- 도메인별 메뉴 노출 여부를 제어한다.
- 단, 최종 권한 판정을 프런트만 믿지 않는다.

### 8.2. 백엔드

- 모든 보호 API에서 최종 권한을 다시 판정한다.
- 도메인별 자원 범위까지 포함해 권한을 확인한다.
- 감사 로그가 필요한 작업은 수행 주체를 기록한다.

---

## 9. 감사와 승인 흐름

- 그룹, 모임, 공지, 공개 콘텐츠 변경은 수행자 `userId`를 기록한다.
- GSR 승인, 정지, 그룹 매핑 같은 권한성 변경은 이력 대상이다.
- 주문, 발송, 만료 처리 같은 운영 액션도 수행자를 남길 수 있어야 한다.

---

## 10. 구현 우선순위

1. `userId`, 계정 상태, 전역 역할을 포함한 최소 로그인 모델 구현
2. 프런트 라우트 보호와 공통 인증 상태 관리
3. `general.admin(service)` 권한 판정과 GSR 범위 제한
4. `store`, `heart` 운영 권한 분리
5. 감사 로그와 승인 흐름 보강

---

## 11. 후속 상세화 항목

- 외부 IdP 최종 선택
- 세션과 JWT 중 기본 전략 확정
- 토큰 갱신 전략
- 권한 변경 시 캐시 무효화 방식
- 운영자 계정 발급과 회수 절차
