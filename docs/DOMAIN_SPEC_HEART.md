# A.A. KOREA 웹앱 도메인 명세서: Heart

공통 아키텍처 원칙과 모듈 간 공통 규칙은 [DOMAIN_SPEC.md](./DOMAIN_SPEC.md)를 따른다.
이 문서는 `heart` 앱 도메인의 구독 모델과 발송 규칙을 정리하며, 내부적으로 `public`과 `admin` 서브도메인을 함께 다룬다.

## 1. 책임 범위

`heart`는 월간지 장기 구독, 발송 이력, 만료 관리, 디지털 아카이브를 담당하는 정기 구독 도메인이다.
향후 `heart.aakorea.org`로 분리 가능한 구조를 목표로 한다.

- `heart.public`: 월간지 소개, 구독 진입, 사용자 구독 조회, 공개 아카이브
- `heart.admin`: 구독, 발송 이력, 만료, 아카이브 운영

이 도메인이 직접 소유하는 핵심 엔티티는 다음과 같다.

- `Subscription`
- `DeliveryHistory`
- `MagazineIssue`

---

## 2. 핵심 기능

### 2.1. 장기 구독

- 1년(12회), 2년(24회) 등 선결제 기반 플랜
- 플랜별 총 발송 횟수 보유
- 구독 시작일, 종료 예정일, 잔여 횟수 계산

### 2.2. 선물 구독

- `buyerUserId`로 결제자를 식별한다.
- 수령인은 A.A. 멤버일 수도 있고 비회원일 수도 있다.
- 교도소, 병원, 시설 등 특수 수령처도 지원해야 한다.

### 2.3. 배송지 변경

- 활성 구독의 다음 발송 대상은 최신 수령인 스냅샷을 따른다.
- 과거 발송 이력은 당시 기준 사실로 유지한다.

### 2.4. 발송 및 만료 관리

- 월별 발송 명단 생성
- `DeliveryHistory` 기록
- `sentCount` 증가
- 필요 시 `EXPIRED` 자동 전환

### 2.5. 디지털 아카이브

- 발행월
- 호수명
- 표지 이미지
- PDF 또는 웹진 링크
- 공개 여부

---

## 3. 도메인 규칙

- 수령인 정보는 `Subscription` 내부 스냅샷으로 관리한다.
- 과거 발송 이력은 `DeliveryHistory`에 별도 사실로 남긴다.
- 발송 횟수는 수동 편집보다 발송 처리 이벤트 기준으로 증가시키는 것이 원칙이다.
- `sentCount >= totalIssues`가 되면 구독 상태를 `EXPIRED`로 전환한다.
- 만료 임박 대상은 잔여 1회 또는 만료 30일 전 기준으로 조회할 수 있어야 한다.

---

## 4. 외부 도메인 계약

### 4.1. `user`

- 구독 생성 시 `buyerUserId`로 결제자를 식별한다.
- 실제 수령인 정보는 `heart`가 자체 스냅샷으로 저장한다.

### 4.2. `heart.public`과 `general.public`

- 디지털 아카이브는 `heart.public` 또는 `general.public(basic)`에서 소비될 수 있다.
- 공개 조회와 구독 상태의 쓰기 책임은 분리한다.

### 4.3. `heart.admin`과 `general.admin`

- 구독, 발송, 아카이브 운영 책임은 `general.admin(service)`가 아니라 `heart.admin`이 가진다.
- 관리자 포털이 통합 UI를 제공하더라도 구독 상태 변경과 발송 기록 생성은 `heart`에서 처리한다.

---

## 5. 대표 유스케이스

1. 관리자가 해당 월 발송 대상을 조회한다.
2. 엑셀 추출 또는 발송 배치를 실행한다.
3. 각 대상마다 `DeliveryHistory`를 생성한다.
4. `sentCount`를 증가시키고 필요 시 `EXPIRED`로 전환한다.

---

## 6. 구현 초점

- 구독 발송 횟수 증가와 만료 전환
- 선물 구독 수령인 스냅샷 고정
- 활성 구독 주소 변경과 과거 발송 이력 분리
- 만료 임박 대상 조회와 알림 배치 조건
